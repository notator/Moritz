SVG Score Extensions

Summary (December 2014)

The extensions are contained in the following namespace:
xmlns:score="http://www.james-ingram-act-two.de/open-source/svgScoreExtensions.html.

These are the extensions written by the latest version of my Assistant Composer software.
I am currently working on the version of the Assistant Performer, which will read them (see the notes below).
These extensions are subject to change without notice!

This version introduces a new type of staff: The Input Staff. The staves in existing scores are all Output Staves.

Output Staves contain Output Chord elements (among other things). Each Output Chord contains a midiChord element containing midi information that is ultimately sent to a midi output device.

Correspondingly, Input Staves contain Input Chords, and each Input Chord contains an inputChord element.
inputChord elements handle midi information coming from a midi input device.
Each inputChord contains a list of inputNote elements, each of which contains a list of seqRef elements, each of which refers to a contiguous, closed sequence of Output Chords and Rests defined in an Output Voice.
A contiguous, closed midi output sequence is called a "Seq" in my software, but this type does not appear explicitly in this file format. (The Assistant Composer constructs an Output Voice from a sequence of Seqs.)

This file definition allows very complex output events to be triggered using single NoteOn input events, but it also allows single inputNotes to trigger single one-note Output Chords — which would be like playing a "prepared piano", except that the preparation can change during the performance.
Different scores can therefore be created for performing the same output information at different levels of control. Recordings are performed by pressing a single button. Piano pieces are traditionally played by pressing a single key for each note...

SVG <g> elements defining container types

Definition: An msDuration is an absolute millisecond duration. msDurations are stored in the score as score:msDuration values, and are are used as the durations in default performances (when there is no live performer).
An msDuration is also used to calculate an object's msPositionInScore (the position in absolute milliseconds from the beginning of the score). Objects in different voices that have the same msPositionInScore are both vertically aligned in space and logically synchronous.

The following SVG <g> elements contain both graphics and special types defined in the score: namespace (see below).
These elements all have a class attribute that will be used by the Assistant Performer to identify their type.
The id attributes are just for convenience while debugging.

class "inputChord"
score:msDuration Integer: The msDuration in the score of this Input Chord.
This value is used by the Assistant Performer to calculate the Input Chord's msPosition, which is the msPosition of the Seqs to which the Input Chord points.
Each Input Chord is has the same msPosition as (and is vertically aligned to) its related Seqs.
Remarks:
1. Input Voices contain no output information, so they are ignored in default performances. This means that Input Chord msDurations are never converted to real milliseconds.
2. It would have been possible to have had an msPosition attribute here, but the Assistant Performer calculates the msPositions of all Output Chords and Rests anyway (so that it can send simultaneous Output Chords in different voices simultaneously). The Assistant Performer software therefore becomes simpler if Input Chords are not treated differently here.

inputControls Optionally contained in either Output Voice or seqRef elements.
This element sets the performance options for a Seq, by individually overriding the current options in the Seq's Output Voice.
Whether the result of the overriding is temporary (just for the Seq) or permanent (set in the Output Voice) is set using the onlySeq option. The options are described in detail below, but each Output Voice initially has the following set of default options:
noteOnKey "ignore"
noteOnVel "ignore"
noteOff "ignore"
pressure "ignore"
pitchWheel "ignore"
modulation "ignore"
speedOption "none"
Attributes:
onlySeq Optional. A 'boolean' integer string. "0" or "1". Default is "0".
If this attribute's value is present (its value will be "1"), all the other options in this inputControls element will override the Output Voice's current settings only for this Seq. Otherwise, the Output Voice's settings are set to the result of the overriding.
The onlySeq option is not itself maintained in the Output Voice.
noteOnKey Optional. A string describing how the key value of a received noteOn is to be used. Default is the current value of this option in the Output Voice.
When a noteOn is received, both its performed key value and its notated key value are sent to the related Seqs, which then decide how to interpret the data.
Values can be:
  • "ignore" - the Seq is performed, but the performed key value is ignored. The key values (pitches) in the output sequence will be as defined in the score.
  • "transpose" - transpose the Seq by the difference between the performed and notated keys.
  • "matchExactly" - only play the Seq if the performed and notated keys match. If they don't match, the performed key is ignored, and the Seq does not play.
noteOnVel Optional. A string describing how the velocity value of a received noteOn is to be used. Default is the current value of this option in the Output Voice.
When a noteOn is received, both its performed velocity value and its notated velocity value are sent to the related Seqs, which then decide how to interpret the data.
Values can be:
  • "ignore" - the Seq is performed, but the performed velocity value is ignored. The velocities in the output sequence will be as defined in the score.
  • "scale" - the velocities in the original output sequence are scaled: If the performed velocity is less than 64, they are reduced. Otherwise they are increased.
noteOff Optional. A string describing what happens when a noteOff is received while the Seq is playing. Default is the current value of this option in the Output Voice.
Values can be:
  • "ignore" - allows the Seq to complete (as written in the score, subject to the noteOn options)
  • "stop" - stop after completing the current Output Chord or Rest.
  • "stopNow" - stop immediately (even while an Output Chord or Rest is playing). Both stop options send a midi allNotesOff message to the channel when complete.
  • "fade" - fades (velocity) the Seq to its end
  • "shortFade" the Seq is faded (velocity) over a fixed number of chords, or to the end of the Seq if that is sooner. The number of objects to fade is then provided in the shortFade attribute (see below).
shortFade An integer. This attribute does not exist unless noteOff is set to "shortFade". If it exists, it sets the number of objects to fade.
shortFade=0 means the same as noteOff="stop".
pressure Optional. A string that is the name of one of the following midi controllers.
Default is the current value of this option in the Output Voice.
If present, the values of received aftertouch or channel pressure messages are converted to messages of the respective type, and sent to the Seq's channel.
Possible values are:
"ignore"
"aftertouch"
"channelPressure"
"pitchWheel"
"modulation"
"volume"
"pan"
"expression"
"timbre"
"brightness"
"effects"
"tremolo"
"chorus"
"celeste"
"phaser"
pitchWheel Optional. A string that is the name of one of the above midi controllers
(see pressure). Default is the current value of this option in the Output Voice.
If present, the values of received pitchwheel messages are converted to messages of the respective type, and sent to the Seq's channel.
modulation Optional. A string that is the name of one of the above midi controllers
(see pressure). Default is the current value of this option in the Output Voice.
If present, the values of received modulationWheel messages are converted to messages of the respective type, and sent to the Seq's channel.
maxVolume Present only if one of the above continuous controllers is set to send volume messages. This is then the maximum MIDI volume value that can be sent to the Output Voice's MIDI channel.
Seqs never send their own volume messages. The Assistant Performer always sends volume messages directly to the Output Voice's MIDI channel.
maxVolume is always greater than minVolume.
minVolume Present only if one of the above continuous controllers is set to send volume messages. This is then the minimum volume value that can be sent.
maxVolume is always greater than minVolume.
speedOption Optional. A string having one of the following values:
"none"
"noteOnKey"
"noteOnVel"
"pressure"
"pitchWheel"
"modulation"
Default is the current value of this option in the Output Voice.
If the value is not "none", the speed of the Seq will vary according to the value of the named controller, using the value of maxSpeedPercent (see below).
maxSpeedPercent Present only if the speedOption is not "none".
An integer greater than 100 denoting the percentage factor by which to divide the notated durations (i.e. go faster) when the controller has its maximum value (127).
The minimum speed (the reciprocal factor) happens when the controller sends a value of 0.

Notes (December 2014)

The score, as currently written by the Assistant Composer:

  • If there are any Input Voices in the score, the Assistant Composer automatically includes all the Output Voices in the file, so that Input Voices always have access to the midi information to which they refer. Output Voices can, however, easily be made invisible.
  • If they are visible, Output Staves are smaller than Input Staves, and are printed above the Input Staves in each System.
  • Input Chords are usually aligned vertically with the first Output Chord in the Seq to which they refer, but it is possible for the Seq to be delayed and so aligned to the right of its triggering Input Chord (see seqRef.msOffset).
  • The same midi channel is never allocated to more than one Output Voice, but this is not true of Input Voices. There may be more than one parallel Input Voice, and Input Voices can share the same midi input channel (i.e. be played by the same polyphonic device). The Assistant Composer can write any number of Input Staves, each of which can contain up to two Input Voices.

Algorithm for the projected Assistant Performer.

This is just a sketch, describing how the Assistant Performer should work in principle. The details may change.
Note that the msPositionInScore is not used as a number of milliseconds in live performances.
It is simply an abstract value that is used as an identifier for sequential events described in the score.
At load time, the following are created:
  • A worker for each output midi channel (=Output Voice) in the score. Each worker is initialized with:
    • a private, ordered list of objects having the fields: {msPositionInScore, seq}. If a seq overlaps the one in the previous object, a load time exception is thrown.
    • a private, ordered list of objects having the fields: {msPositionInScore, score:inputControls} from which the worker can infer its current inputControls settings at runtime. This list need not contain as many entries as in the worker's other list. And the inputControls only need to contain the values that differ from the defaults.
  • In the main thread:
    • The seqRefs contained in each Input Voice are used to construct temporally ordered lists of objects having the fields {msPositionInScore, List<outputChannel>}. There should be one such list per input device (=midi input channel). If more than one Input Voice has the same midi input channel, they are combined in the same list.
    • The threshTime is set. This is a fixed threshold (number of milliseconds) within which noteOns are accepted for a particular msPositionInScore  (see below).

At runtime:
The performing software always listens, and react to, single NoteOn/NoteOff events, relating them to single noteheads that are notated as part of an Input Chord.
Input NoteOn and NoteOff messages arrive in the browser's main thread. The AP knows the current msPositionInScore for each input device, and the midi channel of the incoming event, so it locates the appropriate List<outputChannel> and sends the NoteOn or NoteOff together with its msPositionInScore, to the appropriate output channel workers.
The workers decide what to do with this incoming information depending on the current value of their inputControls.
Incoming midi continuous controller information should only be sent to the input device's currently performing seqs, so:
  • In the main thread:
    • Lists are maintained (one per input device) of the NoteOns and NoteOffs sent to each output channel.
      Continuous controller information is only sent to the tracks that are still playing at least one note (= seq).

Updating the msPositionInScore  per device at runtime:
There will be a fixed threshold (number of milliseconds) within which noteOns are accepted for a particular msPositionInScore. This limit (called threshTime in deference to Max's thresh and quickthresh objects), must be less than the lower msDuration limit of the shortest duration class in the score.
If the score contains fourFlag symbols whose lower limit is 25ms (the standard value in my current scores), the threshTime could be set to, perhaps, 15ms. If the score only contains threeFlag symbols, then threshTime might be set to twice that (30ms). The performer could be given some choice here.

When the threshTime expires for a particular input device in the main thread, or the correct number of noteOns has arrived, noteOn handling could be turned off for that device until the score gets close to its next msPositionInScore.
NoteOff messages, however, would continue to be handled for NoteOns that have been sent, and sent to the channels associated with their corresponding noteOn.

Inside workers (at runtime):
Depending on the current inputControls settings, incoming NoteOff messages may or may not affect a currently playing seq. Similarly, incoming NoteOn messages may be ignored if their key value does not match an expected one at that msPositionInScore, or if all the seqs at that position have already been started. If the noteOn for a particular seq never arrives (it might not be played at all, or it might arrive too late from the input device), the seq will never be played.
